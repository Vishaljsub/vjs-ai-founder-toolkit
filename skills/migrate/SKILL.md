# description: Create and apply database migrations for {{APP_NAME}}, template

Copy into `.claude/skills/migrate/SKILL.md` and fill in the `{{...}}` placeholders. Written for
Alembic (Python/SQLAlchemy); adapt the commands if your stack uses a different migration tool
(Prisma, Rails, Django, Flyway, etc.) — the workflow shape stays the same.

## Apply pending migrations (run inside the backend container)
```bash
docker compose exec {{BACKEND_SERVICE}} alembic upgrade head
```

## Create a new migration after model changes
```bash
docker compose exec {{BACKEND_SERVICE}} alembic revision --autogenerate -m "describe_change_here"
# Then review the generated file in {{MIGRATIONS_DIR}}/ before applying
docker compose exec {{BACKEND_SERVICE}} alembic upgrade head
```

## Run locally (if running the backend outside Docker)
```bash
cd {{BACKEND_DIR}}
alembic upgrade head
alembic revision --autogenerate -m "describe_change"
```

## Check current migration state
```bash
docker compose exec {{BACKEND_SERVICE}} alembic current
docker compose exec {{BACKEND_SERVICE}} alembic history --verbose
```

## Rollback one step
```bash
docker compose exec {{BACKEND_SERVICE}} alembic downgrade -1
```

## Important rules
- Always review autogenerated migrations before applying — autogenerate misses some column/type
  changes and index renames; check the diff by hand.
- **Never delete a migration file that has already been applied anywhere** (local, staging, or
  prod) — it breaks the migration chain for anyone re-running from an earlier state.
- If migrations run automatically on app startup (common for zero-touch deploys), make sure only
  a **single instance** runs the migration step to avoid two processes racing on the same schema
  change — see the `deploy-gcp-cloudrun` skill's "migrations racing" gotcha for one way to handle
  this on serverless platforms.
